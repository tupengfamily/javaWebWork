## Context

当前系统已接入 6 个数据源（起点、番茄、纵横、豆瓣、百度、当当），`novel` 表数据量约 1000 条，`ranking_record` 表约 60 条且随爬取次数持续增长。

现有「数据管理」页面（`CrawlerData.vue`）的问题：
1. **站点状态卡片与任务管理页重复** — 两个页面都调用 `dashboardSites()` API，展示相同的站点状态
2. **缺乏数据质量视图** — 管理员无法快速识别缺封面/缺简介/缺作者的脏数据
3. **操作效率低** — 只支持单条删除，不支持批量操作
4. **无导出能力** — 无法将数据导出为 CSV 进行离线分析

后端技术栈：Spring Boot 3 + MyBatis-Plus + MySQL 8，前端：Vue 3 + Element Plus + Pinia。

## Goals / Non-Goals

**Goals:**
- 提供数据质量仪表盘，让管理员一眼看到各站点数据完整性
- 支持批量删除和 CSV 导出，提升管理效率
- 增强小说详情弹窗（排行记录趋势迷你图）
- 保持页面职责清晰：任务管理 = 运维，数据管理 = 数据治理

**Non-Goals:**
- 不做数据手动编辑（会被爬虫覆盖）
- 不做数据备份/恢复（基础设施层职责）
- 不做数据对账/校验自动化（当前规模不需要）
- 不做数据迁移或 schema 变更

## Decisions

### 1. 数据质量统计 — 后端 SQL 聚合 vs 前端全量计算

**选择：后端 SQL 聚合**

后端直接用 `GROUP BY` + `COUNT` + `CASE WHEN` 统计每个站点的数据完整性指标，前端只渲染结果。

**理由：**
- 数据量增长后前端全量计算不可持续
- SQL 聚合在 MySQL 中极快（<10ms）
- API 返回结构化 JSON，前端无需复杂逻辑

**替代方案：** 前端翻页遍历所有小说计算 — 不可行，数据量大时请求过多。

### 2. CSV 导出 — 后端生成 vs 前端 Blob

**选择：后端生成**

后端直接查询数据并写入 CSV 流，设置 `Content-Type: text/csv` + `Content-Disposition: attachment`。

**理由：**
- 数据量可能超过前端分页限制，后端可一次导出全量
- 避免前端内存压力
- 浏览器原生支持下载

**替代方案：** 前端将当前页数据转 Blob — 限制太多，只能导出当前页。

### 3. 批量删除 — 逐条调用 vs 后端批量接口

**选择：后端批量接口**

新增 `DELETE /api/admin/novels/batch` 接口，接收 ID 列表，单事务内批量删除。

**理由：**
- 避免 N 次 HTTP 请求
- 单事务保证原子性
- MyBatis-Plus 的 `deleteBatchIds` 直接可用

### 4. 页面布局重构 — 保留 vs 重写

**选择：重写**

移除站点状态卡片（与任务管理重复），改为 3 个 Tab：数据质量概览、小说管理、排行记录。

**理由：**
- 站点状态卡片在任务管理页已有，此处重复
- Tab 切换比滚动更清晰
- 数据质量概览需要首屏位置

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| CSV 导出大数据量时 OOM | 限制导出上限 10000 条，使用流式写入 |
| 批量删除误操作 | 前端二次确认弹窗，后端事务回滚 |
| 数据质量统计频率过高 | API 响应设置 Cache-Control: max-age=60 |
| 排行记录趋势迷你图增加前端复杂度 | 使用轻量 SVG 折线图，不引入 ECharts |
